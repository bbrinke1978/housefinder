<objective>
Build the configurable distress scoring engine that calculates weighted scores from active signals and flags hot leads. This runs after each scrape to keep lead scores current.

Purpose: Scoring is the intelligence layer that turns raw scraped data into actionable lead prioritization. Without it, all properties look the same -- the investor cannot distinguish a property with one stale tax lien from one with a fresh NOD plus active lis pendens.

Output: A pure scoring function (testable without DB), a database-aware orchestrator that reads config and updates leads, and a seed script for default configuration values.
</objective>

1. Create `scraper/src/scoring/types.ts`:
   ```typescript
   export interface SignalConfig {
     signal_type: string;
     weight: number;
     freshness_days: number; // signals older than this are excluded from score
   }

   export interface ScoringConfig {
     signals: SignalConfig[];
     hot_lead_threshold: number;
   }

   export interface ScoreResult {
     score: number;
     is_hot: boolean;
     active_signal_count: number;
     scored_signal_count: number; // signals that passed freshness check
   }

   export interface SignalInput {
     signal_type: string;
     recorded_date: Date | null;
     status: string;
   }
   ```

2. Create `scraper/src/scoring/score.ts` with TWO exported functions:

   **Pure function (no DB dependency -- testable):**
   ```
   export function scoreProperty(signals: SignalInput[], config: ScoringConfig): ScoreResult
   ```
   - Filter to active signals only (status === 'active')
   - For each active signal, find its config by signal_type
   - If no config for that signal_type, skip it (allows new signal types to be added without crashing)
   - Check freshness: if recorded_date is null, include it (no date = assume recent). If recorded_date is set, compute age in days. If age > freshness_days, exclude from weighted score.
   - Sum weights of all qualifying signals
   - Return { score, is_hot: score >= config.hot_lead_threshold, active_signal_count, scored_signal_count }

   Research recommendation on weights (Claude's discretion per CONTEXT.md):
   - NOD = 3 (highest urgency, ~90 day auction window in Utah trust deed states)
   - tax_lien = 2 (serious but redeemable)
   - lis_pendens = 2 (active litigation, uncertain resolution)
   - probate = 1 (future use, Phase 4)
   - code_violation = 1 (future use, Phase 4)
   - vacant = 1 (future use, Phase 4)
   - hot_lead_threshold = 4 (NOD alone = 3, not hot; NOD + tax_lien = 5, hot)
   - NOD freshness_days = 75 (before auction window closes)
   - tax_lien freshness_days = 365 (annual cycle)
   - lis_pendens freshness_days = 180 (litigation can drag)

   **Database orchestrator:**
   ```
   export async function scoreAllProperties(): Promise<{ scored: number; hot: number }>
   ```
   - Read scoring config from scraperConfig table (keys: 'scoring_signals', 'hot_lead_threshold')
   - Parse config values from JSON strings
   - Query all properties with their active distress signals (JOIN properties + distressSignals WHERE status = 'active')
   - For each property, call scoreProperty() with its signals and parsed config
   - Upsert to leads table: set distressScore, isHot, update updatedAt. If no lead row exists for the property, create one with status 'new' and newLeadStatus 'new'.
   - Return count of scored properties and hot leads
  </action>
