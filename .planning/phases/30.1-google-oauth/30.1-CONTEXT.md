# Phase 30.1: Google Workspace OAuth Login — Context

**Gathered:** 2026-04-29
**Status:** Light spec — ready to execute
**Source:** Direct conversation with Brian after creating Stacee with a temp password (the temp-password dance is the friction this phase eliminates)

## Goal

Replace the temp-password user-onboarding dance with "Sign in with Google" using the existing Google Workspace `@no-bshomes.com` accounts. After this phase ships, creating a new team member becomes:

1. Brian creates `<name>@no-bshomes.com` mailbox in Google Workspace (he does this anyway for email)
2. New user clicks "Sign in with Google" on `/login`, authenticates with their Workspace credentials
3. NextAuth auto-creates a row in `users` with empty roles
4. Brian sees them as "Pending" in `/admin/users`, assigns roles
5. User is in

No admin scripts, no temp passwords, no `/forgot-password` round-trips.

## Locked decisions

### Provider strategy

- **Add** Google OAuth as a NextAuth provider alongside the existing `Credentials` provider. Don't remove Credentials — Brian, Shawn, admin@ keep email/password as a fallback. Stacee (created today via temp password) keeps her current login until she chooses to use Google.
- The login page shows the Google button **prominently on top** with the email/password form below it under a small "Or sign in with email" header.

### Domain restriction

Same `@no-bshomes.com` rule as the existing Credentials provider. NextAuth's `signIn` callback rejects any Google profile whose email doesn't end with `@no-bshomes.com`. This works because the OAuth consent screen's audience is set to "Internal" in Google Cloud Console (Workspace-only restriction at Google's layer), but we belt-and-suspenders at the NextAuth layer too.

### Auto-provisioning on first login

When a Workspace user signs in via Google for the first time and no DB row matches their email:

- INSERT a new `users` row with: email (from Google), name (from Google profile), `password_hash = ''` (empty — never used; Google login bypasses bcrypt), `roles = []`, `is_active = true`.
- Login proceeds, but the existing role-empty gate (Phase 29's `roles.length === 0 → reject`) blocks the session from being established.

Wait — that's a contradiction. If we reject sessions when roles are empty, the auto-provisioned user can't get past login. Two options:

| Option | Behavior |
|---|---|
| **A: Reject at login** | Session is never created. User sees a generic login error. Brian has to find them in `/admin/users` (where the row now exists) and grant roles before they can try again. |
| **B: Allow login + render "Pending approval" page** | Session is created with empty roles. The dashboard middleware redirects empty-role users to `/pending-approval` page that explains "Your account is awaiting Owner approval". Brian sees them in `/admin/users` AND the new user can refresh the page once approved. |

**Decision: Option B.** Better UX — the new user gets feedback that their login worked, they're just waiting for role assignment. The middleware redirect is one new file. Brian's "pending users" view in `/admin/users` already exists from Phase 30.

### Existing-user matching

NextAuth's standard behavior: when a user signs in via Google with email X, and a `users` row already exists with email X, the existing row is reused. No duplicate is created. Brian + Shawn + admin@ + Stacee all keep their current rows; they can switch between Google login and password login interchangeably.

### Login page

- Big "Sign in with Google" button at top (Google's brand-compliant button styling — square, white background, Google "G" icon, the words "Sign in with Google")
- Divider: "Or sign in with email"
- Existing email/password form below

### Google Cloud Console setup

Manual one-time setup Brian does in his browser:

1. Create or open a Cloud project named "No BS Homes" or similar
2. APIs & Services → OAuth consent screen → set audience to **Internal** (this is the magic: only `@no-bshomes.com` accounts can complete the flow)
3. APIs & Services → Credentials → Create Credentials → OAuth client ID → type "Web application"
4. Authorized JavaScript origin: `https://finder.no-bshomes.com`
5. Authorized redirect URI: `https://finder.no-bshomes.com/api/auth/callback/google`
6. Copy the Client ID + Client Secret
7. Add to Netlify env vars as `GOOGLE_CLIENT_ID` and `GOOGLE_CLIENT_SECRET`
8. Add the same to `app/.env.local` for local dev

Documented in `docs/google-services.md` section 3.

### Out of scope (deferred)

- 2FA on local Credentials login (Google sign-in already gets MFA from Workspace policy — that's enough for now)
- Migrating Brian/Shawn/admin@ to Google-only (they keep both options)
- Auto-grant roles based on Google Group membership (interesting future direction; defer)
- Other OAuth providers (GitHub, Microsoft, etc.)
- Social login for the public marketing site `no-bshomes.com` (different app)
