# Phase 30.1 Technical Research

## Summary

Confidence: **HIGH**. NextAuth has a first-party Google provider (`next-auth/providers/google`) that handles the OAuth dance. The domain restriction + auto-provisioning logic goes in the existing `signIn` and `jwt` callbacks. Total work: small.

No new dependencies (NextAuth core is already in `app/package.json`). No schema changes (existing `users` table works as-is — `password_hash` becomes optional/empty for Google-provisioned users).

2. Top bar → project dropdown → "New Project". Name it something like "No BS Homes" or "no-bshomes-app".

8. Application type: **Web application**.
9. Name: "No BS Workbench Web App" (or anything).

    - `https://finder.no-bshomes.com`

    - `https://finder.no-bshomes.com/api/auth/callback/google`

## Code changes

### 1. NextAuth config — `app/src/auth.ts`

Add Google provider alongside the existing Credentials provider. Add a `signIn` callback to enforce domain restriction and handle auto-provisioning. Extend the `jwt` callback to load roles for Google sign-ins (the existing logic only sets roles on initial Credentials sign-in via the `user` parameter).

```typescript
import NextAuth from "next-auth";
import Credentials from "next-auth/providers/credentials";
import Google from "next-auth/providers/google";

export const { handlers, auth, signIn, signOut } = NextAuth({
  providers: [
    Google({
      clientId: process.env.GOOGLE_CLIENT_ID!,
      clientSecret: process.env.GOOGLE_CLIENT_SECRET!,
    }),
    Credentials({
      // ... existing implementation unchanged
    }),
  ],
  callbacks: {
    async signIn({ user, account }) {
      // Google provider: enforce @no-bshomes.com domain + auto-provision
      if (account?.provider === "google") {
        const email = (user.email ?? "").toLowerCase();
        if (!email.endsWith("@no-bshomes.com")) {
          return false; // reject — domain restriction
        }

        const { db } = await import("@/db/client");
        const { users } = await import("@/db/schema");
        const { eq } = await import("drizzle-orm");

        const [existing] = await db
          .select({ id: users.id, isActive: users.isActive })
          .from(users)
          .where(eq(users.email, email))
          .limit(1);

        if (!existing) {
          // Auto-provision: insert new user with empty roles. Brian assigns
          // roles via /admin/users; until then, the user lands on /pending-approval.
          await db.insert(users).values({
            email,
            name: user.name ?? email.split("@")[0],
            passwordHash: "", // never used — Google login bypasses bcrypt
            roles: [],
            isActive: true,
          });
        } else if (!existing.isActive) {
          return false; // deactivated user — block re-entry via Google too
        }
      }
      return true;
    },
    async jwt({ token, user, account }) {
      // On initial sign-in (any provider), load fresh user data from DB so
      // the JWT carries the latest roles + id even after auto-provisioning.
      if (user || account?.provider === "google") {
        const email = (token.email ?? user?.email ?? "").toLowerCase();
        if (email) {
          const { db } = await import("@/db/client");
          const { users } = await import("@/db/schema");
          const { eq } = await import("drizzle-orm");
          const [row] = await db
            .select({ id: users.id, roles: users.roles })
            .from(users)
            .where(eq(users.email, email))
            .limit(1);
          if (row) {
            token.userId = row.id;
            token.roles = row.roles ?? [];
          }
        }
      }
      return token;
    },
    async session({ session, token }) {
      // Existing logic — unchanged
      if (session.user) {
        (session.user as { id?: string }).id = token.userId as string;
        (session.user as { roles?: string[] }).roles = (token.roles as string[]) ?? [];
      }
      return session;
    },
  },
});
```

**Key differences from current auth.ts:**

- The existing `Credentials.authorize` callback rejects users with empty roles. **The Google flow does NOT reject empty-roles users at signIn time** — they're allowed to authenticate, but the middleware (next change) redirects them to `/pending-approval`.
- The `jwt` callback now reads from DB on every sign-in (not just when `user` is present). This is needed because Google sign-ins bypass `authorize`, so we need a separate path to attach the roles.

### 2. Middleware — `app/src/middleware.ts` (new file or extend existing)

Add a redirect for users with empty roles (Google-provisioned but not yet approved):

```typescript
import { auth } from "@/auth";
import { NextResponse } from "next/server";
import type { NextRequest } from "next/server";

export async function middleware(req: NextRequest) {
  const session = await auth();
  if (!session?.user) {
    // Not authenticated — let the existing public-route logic handle it
    return NextResponse.next();
  }

  const roles = ((session.user as { roles?: string[] }).roles) ?? [];
  const path = req.nextUrl.pathname;

  // Allow the pending-approval page itself, /api routes, /login, and static assets
  const allowedPathsForEmptyRoles = ["/pending-approval", "/login", "/api", "/forgot-password", "/reset-password"];
  const isAllowed = allowedPathsForEmptyRoles.some((p) => path === p || path.startsWith(p + "/"));

  if (roles.length === 0 && !isAllowed && !path.startsWith("/_next")) {
    return NextResponse.redirect(new URL("/pending-approval", req.url));
  }

  return NextResponse.next();
}

export const config = {
  matcher: [
    // Match all routes EXCEPT static assets
    "/((?!_next/static|_next/image|favicon.ico).*)",
  ],
};
```

If `app/src/middleware.ts` already exists (NextAuth provides one by default), merge this logic into it.

### 3. Pending-approval page — `app/src/app/pending-approval/page.tsx`

```tsx
import { auth, signOut } from "@/auth";
import { redirect } from "next/navigation";

export default async function PendingApprovalPage() {
  const session = await auth();
  if (!session?.user) redirect("/login");

  const roles = ((session.user as { roles?: string[] }).roles) ?? [];
  if (roles.length > 0) redirect("/"); // already approved — kick them to dashboard

    <div className="flex min-h-screen items-center justify-center bg-background p-4">
      <div className="max-w-md text-center space-y-4">

        <p className="text-sm text-muted-foreground">

          your account hasn't been assigned a role yet. The Owner needs to grant

        <p className="text-xs text-muted-foreground">
          Once your role is assigned, refresh this page and you'll be in.
        </p>
        <form action={async () => { "use server"; await signOut({ redirectTo: "/login" }); }}>
          <button
            type="submit"
            className="text-sm text-primary hover:underline"
          >
            Sign out
          </button>

      </div>

  );
}
```

## Schema changes

**None.** The existing `users` table works:

- `email` — populated from Google profile
- `name` — populated from Google profile (e.g. "Stacee Smith")
- `password_hash` — set to `''` (empty string) for Google-only users; bcrypt's `compare` against `''` always returns false, so a Credentials login attempt with an empty hash would never match. Safe.
- `roles` — empty array for newly auto-provisioned users; Owner grants via `/admin/users`
- `is_active` — defaults to `true`

A future hardening: add a nullable `password_hash` and rely on NULL-check in the Credentials authorize. Defer — not blocking.

## Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| Brian misconfigures the OAuth consent screen as "External" instead of "Internal" | Medium | The signIn callback enforces the domain restriction at the NextAuth layer regardless. Belt + suspenders. |
| `GOOGLE_CLIENT_SECRET` accidentally committed to git | Medium-low | `.env.local` is gitignored. Never reference the secret literal in source files. Phase shipped → rotate the secret if leaked (Cloud Console → Reset secret). |
| Google profile email differs in case from existing DB row | Low | `signIn` callback lowercases everything. The DB email column is also lowercase by convention. |
| Existing user logs in via Google and we end up with two rows | Low | NextAuth's email-match prevents this — it reuses the existing row when emails match. Verified by the `if (!existing)` branch that only INSERTs when no row exists. |
| Auto-provisioned user with roles=[] gets stuck in a redirect loop | Low | Middleware exempts `/pending-approval`, `/login`, `/api`, `/forgot-password`, `/reset-password`. If a loop happens, sign-out is always available. |
