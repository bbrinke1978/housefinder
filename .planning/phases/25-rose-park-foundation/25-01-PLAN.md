Make three changes:

**Change 1 — Add normalizeCity() function** immediately before the upsertProperty() function definition. The function has a JSDoc comment explaining the zip-to-neighborhood pattern and future expansion:

```typescript
/**
 * Normalizes city name based on zip code for multi-neighborhood urban areas.
 * zip='84116' -> 'Rose Park' (Salt Lake City neighborhood)
 * Future: 84104 -> 'Glendale', 84106 -> 'Sugar House', etc.
 * Gracefully degrades: rural counties without zip pass through unchanged.
 */
function normalizeCity(city: string, zip?: string): string {
  if (zip === '84116') return 'Rose Park';
  return city;
}
```

**Change 2 — Update COUNTY_DEFAULT_CITY map** by adding `"salt lake": "Rose Park"` as a new entry. This covers upsertProperty() for future SLC scrapers that don't set zip.

**Change 3 — Update COUNTY_CITY inline map inside upsertFromUtahLegals()** (the map used by utah-legals notices) by adding `"salt lake": "Rose Park"`. This is a separate map from COUNTY_DEFAULT_CITY — both must be updated or utah-legals SLC notices will fall through to city="".

**Change 4 — Wrap the city resolution on line 67** (currently `const city = record.city || COUNTY_DEFAULT_CITY[resolvedCounty] || "";`) to call normalizeCity:
```typescript
const city = normalizeCity(record.city || COUNTY_DEFAULT_CITY[resolvedCounty] || "", record.zip);
```

Do NOT add salt lake to utah-legals.ts TARGET_COUNTIES — that is Phase 26+ (RP-FW-01, deferred). The upsert layer change prepares the code for when SLC is eventually activated, but does not activate it today.
