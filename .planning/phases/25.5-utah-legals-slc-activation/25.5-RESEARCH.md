## Summary

Phase 25.5 has three implementation tasks, all confined to two files: `scraper/src/sources/utah-legals.ts` and `scraper/src/lib/upsert.ts`. No new npm packages, no schema changes, no Azure Function changes.

**The single most important correction from prior research (v1.3 SUMMARY.md):** Salt Lake County parcel IDs are NOT all-numeric 10-digit (e.g. `2818207018`). Real SLC NOD documents show hyphenated multi-segment format: `26-24-406-084-0000` and `08-22-327-004-0000`. The format is XX-XX-XXX-XXX-XXXX (or similar variable segmentation). The existing `extractParcelId()` regex will NOT match these — it only handles the Carbon County `\d{2}-\d{4}-\d{4}` pattern. A new alternation branch is required.

**Key confirmed facts (HIGH confidence from code):**
- `TARGET_COUNTIES` is `Array<{ index: number; name: string }>`. Adding SLC is one object literal in that array.
- The `extractParcelId()` function has three ordered branches: APN label → Parcel/Serial label → Carbon bare pattern. The SLC format needs to be added as a fourth branch (or widen the APN/label branches since SLC notices will have label prefixes).
- `upsertFromUtahLegals()` already has `"salt lake": "Rose Park"` in its inline `COUNTY_CITY` map (added in Phase 25-01). City normalization is already wired. No change needed there.
- `normalizeCity()` is already live and will retag any SLC property with `zip='84116'` to `city='Rose Park'` automatically. But zip must flow into the `PropertyRecord` for this to trigger — currently `upsertFromUtahLegals()` does not set `zip` on the record it passes to `upsertProperty()`.
- The zip allowlist (RP-11) cannot use `record.zip` reliably from the notice data — zip does not appear as a discrete field in the scraper's notice object. The filter must use a different strategy: filter on the notice's city string matching known 84116 city names, OR extract zip from the address text, OR apply the filter at the post-scrape county level (accept all SLC notices that match `city='Salt Lake City'` or `city='Rose Park'`, reject Sandy/Midvale/etc.).

**Primary recommendation:** Add SLC at index 17, add a new bare-pattern regex branch for the multi-segment SLC format (`\b(\d{2}-\d{2}-\d{3}-\d{3}-\d{4})\b` or a looser variant), implement the zip allowlist by checking the notice's city against a 84116 city allowlist (`['salt lake city', 'rose park', 'north salt lake']`), and pass `zip: '84116'` explicitly on the upsert call when the allowlist passes — so `normalizeCity()` can retag it.

### Pattern 2: extractParcelId() Extension (RP-10)

**What:** Add a regex branch that captures the SLC multi-segment hyphenated format.

**Existing function (from file, HIGH confidence):**
```typescript
function extractParcelId(text: string): string | undefined {
  // Branch 1: A.P.N. label
  const apn = text.match(/A\.P\.N\.?:?\s*([A-Z0-9][\w\-./]{3,})/i);
  if (apn) return apn[1].trim();

  // Branch 2: Parcel No / Serial No / Tax ID label
  const parcelNo = text.match(/(?:Parcel|Serial|Tax ID|Parcel No\.?|Serial No\.?)[:\s#]+([A-Z0-9][\w-]{3,})/i);
  if (parcelNo) return parcelNo[1].trim();

  // Branch 3: Carbon bare pattern XX-XXXX-XXXX
  const carbonParcel = text.match(/\b(\d{2}-\d{4}-\d{4})\b/);
  if (carbonParcel) return carbonParcel[1];

  return undefined;
}
```

**Key finding:** Branches 1 and 2 already capture labeled parcel IDs. If SLC NOD notices include a label like `Serial No: 26-24-406-084-0000` or `Tax Serial Number: 26-24-406-084-0000`, they will be caught by branch 2. The existing `[\w-]{3,}` capture group will grab the full hyphenated value including the trailing segment.

**Gap:** If a notice has no label and only a bare multi-segment number in the text (unlikely but possible), branch 3 only matches `\d{2}-\d{4}-\d{4}` (Carbon format). SLC format has more segments.

**SLC parcel format confirmed from real NOD documents (MEDIUM confidence — from web search metadata citing real documents, not from live Utah Legals SLC notices):** `26-24-406-084-0000` — 5 segments: `DD-DD-DDD-DDD-DDDD`. Also seen: `08-22-327-004-0000`. The SLC assessor confirms format is "6-14 digits" excluding separators. This yields 14 numeric digits + hyphens.

**Recommended fourth branch:**
```typescript
  // Branch 4: SLC-style multi-segment: DD-DD-DDD-DDD-DDDD or similar
  // Covers Salt Lake County bare parcel numbers in notice text without a label prefix.
  const slcParcel = text.match(/\b(\d{2}-\d{2,3}-\d{3,4}-\d{3,4}-\d{3,4})\b/);
  if (slcParcel) return slcParcel[1];
```

**Alternative approach:** Since branches 1 and 2 will likely catch labeled SLC parcels, branch 4 may be unnecessary in practice. The planner should include it for defense but the test run will reveal if branch 4 ever fires.

**MUST verify during planning:** Inspect at least 5 actual SLC NOD notice snippets from Utah Legals to confirm: (a) whether a label prefix is present and which one, (b) exact format of the parcel number as it appears in the snippet. This cannot be done without a logged-in browser session on the live site.

### Pattern 3: 84116 Zip Allowlist Filter (RP-11)

**What:** After SLC notices are scraped, filter to only retain those whose property city resolves to the 84116 zip code area before calling `upsertFromUtahLegals()`.

**Why needed:** Utah Legals returns ALL SLC NODs when SLC is selected — Sandy, Midvale, Holladay, Taylorsville, Murray, West Valley City, etc. Without this filter, the first production run will insert hundreds of non-Rose Park SLC properties. The dashboard will flood with out-of-scope leads.

**Where to apply (HIGH confidence from code):** The filter must be applied in `utah-legals.ts` AFTER the county/foreclosure filtering but BEFORE the call to `upsertFromUtahLegals()`. The upsert function is a generic multi-county handler and should remain zip-agnostic. The correct location is in the same notice processing loop, alongside the existing `isTargetCounty` and `isForeclosure` checks.

**What data is available at filter time:** Each notice has `notice.city` (parsed from the `.right` cell — e.g. `"SALT LAKE CITY"`, `"ROSE PARK"`, `"SANDY"`, `"MIDVALE"`). This is the most reliable field to filter on.

**Allowlist strategy:** The zip 84116 covers Rose Park, parts of North Salt Lake, and the western SLC neighborhood strip. The city values that will appear in Utah Legals SLC notices for 84116 properties are:
- `"SALT LAKE CITY"` — most Rose Park notices will show this (Rose Park is a neighborhood of SLC)
- `"ROSE PARK"` — possible, if the notice uses neighborhood name
- `"NORTH SALT LAKE"` — 84054 bleed into 84116 per SUMMARY.md; could be included or excluded

**Cities to EXCLUDE:** `"SANDY"`, `"MIDVALE"`, `"HOLLADAY"`, `"TAYLORSVILLE"`, `"MURRAY"`, `"MILLCREEK"`, `"WEST VALLEY CITY"`, `"WEST JORDAN"`, `"SOUTH JORDAN"`, `"DRAPER"`, `"HERRIMAN"`, `"RIVERTON"`, `"BLUFFDALE"`, `"COTTONWOOD HEIGHTS"`, `"MAGNA"`, `"KEARNS"`, `"GRANITE"`.

**Problem with city-based allowlist:** If a notice lists city as `"SALT LAKE CITY"` but the property is actually in Sandy (some mailing address discrepancies), we'd incorrectly include it. The inverse is also true — a Rose Park property with a bad city label would be excluded.

**Better approach:** Allowlist-by-zip extracted from address text. But as noted, zip rarely appears in the 300-char snippet.

**Pragmatic approach (MEDIUM confidence):** For phase 25.5, use city-name allowlist: accept `"SALT LAKE CITY"` and `"ROSE PARK"` for SLC county; reject everything else. This will be directionally correct for the majority of cases. The `normalizeCity()` function then further maps to `'Rose Park'` only when zip is `'84116'`. Since we don't have zip from the notice, call `normalizeCity` with the COUNTY_CITY default `"Rose Park"` — which means ALL accepted SLC notices will be tagged `city='Rose Park'` regardless of whether the actual property is in 84116.

**Simpler alternative (recommended by this research):** Accept all SLC NOD notices but let `normalizeCity()` do the work. The function currently maps `zip='84116'` → `'Rose Park'` and passes everything else through. If we don't pass a zip, the city falls through to `COUNTY_CITY["salt lake"] = "Rose Park"` which tags ALL SLC notices as Rose Park. This is incorrect — it includes Sandy properties tagged as Rose Park.

**Therefore the filter IS necessary.** Implementation pattern:

```typescript
// In utah-legals.ts, inside the notice processing loop, after isForeclosure check:
// For SLC notices: only retain those whose city is in the 84116 allowlist.
// SLC returns notices for all zip codes — Sandy, Midvale, Holladay etc. are excluded.
const SLC_CITY_ALLOWLIST = ['salt lake city', 'rose park', 'north salt lake'];

if (notice.county === 'salt lake') {
  const noticeCity = (notice.city ?? '').toLowerCase().trim();
  if (!SLC_CITY_ALLOWLIST.includes(noticeCity)) {
    console.log(`${tag} SLC notice skipped (city not in 84116 allowlist): ${notice.city} (ID: ${result.noticeId})`);
    continue;
  }
}
```

**Then pass zip explicitly when upserting SLC notices:**
In `upsertFromUtahLegals()`, or by enriching the `UtahLegalsNotice` object with a `zip?: string` field before the upsert call, pass `zip: '84116'` for SLC notices that pass the allowlist — so `normalizeCity()` tags them correctly even if the city string is `"SALT LAKE CITY"` rather than `"ROSE PARK"`.

**Alternative simpler approach:** Extend `UtahLegalsNotice` with `zip?: string`, set it to `'84116'` for SLC notices that pass the allowlist, and pass it through to `upsertProperty()` via `upsertFromUtahLegals()`. This uses the existing `normalizeCity()` hook cleanly. This is the recommended approach — it's the minimal change that makes the existing `normalizeCity()` machinery work for SLC.

### Anti-Patterns to Avoid

- **Assuming SLC parcel IDs are all-numeric 10-digit** — Prior research stated this. Real NOD documents show `26-24-406-084-0000` (hyphenated). The existing branch 2 regex (`[\w-]{3,}`) likely captures this correctly IF a label is present. Verify with actual notice text.

## Common Pitfalls

### Pitfall 1: Wrong County Index — Index 17 vs 18
**What goes wrong:** Using index 17 selects "Salt Lake/ Utah" (combined) instead of "Salt Lake" alone — OR "Salt Lake" (index 17) triggers the wrong checkbox. The Playwright `__doPostBack` fires on the wrong county and results come back empty or mixed.
**Why it happens:** The live DOM showed both "Salt Lake" (17) and "Salt Lake/ Utah" (18) as distinct checkboxes. Alphabetical ordering suggests 17 is Salt Lake County only, but this was confirmed via rendered text not raw HTML.
**How to avoid:** Add a `console.log` after the SLC checkbox is checked showing how many notices came back and what county values appear in the results. If count is 0 or county shows "utah", the index is wrong.
**Warning signs:** `[utah-legals] Page 1: found 0 notices` after adding index 17, or county field showing "utah" or "utah county" in notice results.
**Confidence:** LOW until Playwright runtime confirms. MUST verify during planning/test run.

### Pitfall 2: SLC Parcel IDs Null — All Notices Get Synthetic ul- IDs
**What goes wrong:** `extractParcelId()` returns `undefined` for all SLC notices. Every SLC property gets a `ul-salt lake-<address>` or `ul-salt lake-id<N>` synthetic parcel ID. Signal stacking breaks — same property filing next week creates a new DB row.
**Why it happens:** Branch 3 only matches Carbon format `\d{2}-\d{4}-\d{4}`. SLC format is `\d{2}-\d{2}-\d{3}-\d{3}-\d{4}`. Branch 2 would catch a labeled parcel IF the label is one of: Parcel, Serial, Tax ID, Parcel No., Serial No. The exact label used in SLC NOD notice text is unconfirmed.
**How to avoid:** Log `notice.parcelId` for every SLC notice in the test run. If any are null, inspect the raw `snippet` text to see what label (if any) is used and adjust the regex.
**Warning signs:** All SLC notices logging `parcel:none` in the `[utah-legals] Notice ID:...` console line.
**Confidence:** MEDIUM — branch 2 likely catches most but NOT confirmed from real SLC snippet text.

### Pitfall 3: City Allowlist Too Narrow — Valid Rose Park Notices Skipped
**What goes wrong:** A Rose Park property notice shows city as `""` (empty), `"84116"`, or some variant not in the allowlist. The filter excludes it.
**Why it happens:** The city field is parsed from the hidden `.right` cell text using `City:\s*([^\n]+)`. If the cell is missing or the format differs for SLC notices, city will be empty. Utah Legals may not always populate city for all notices.
**How to avoid:** In the filter, treat an empty city as a pass-through for SLC (log a warning but don't exclude). An empty city will get the COUNTY_CITY default `"Rose Park"` and `normalizeCity()` will set it correctly if zip is passed.
**Warning signs:** `[utah-legals] SLC notice skipped (city not in 84116 allowlist): (empty)` appearing in logs.

### Pitfall 4: normalizeCity() Doesn't Fire Because zip Is Not Set
**What goes wrong:** SLC notices land in the DB with `city='Rose Park'` (from COUNTY_CITY default) but `zip` is null. This is actually correct behavior — `normalizeCity()` returns `'Rose Park'` when zip is `'84116'` OR when the county default provides it. Since `COUNTY_CITY["salt lake"] = "Rose Park"`, the city will be `"Rose Park"` regardless of zip. The zip column in the DB just won't have `'84116'` set.
**Why it matters:** Phase 26 UGRC enrichment does NOT use zip to filter — it joins by `parcel_id`. So missing zip on the property row does NOT break UGRC enrichment. But if a future query filters by `zip='84116'`, those SLC rows won't match.
**How to avoid:** Pass `zip: '84116'` explicitly on the `upsertProperty` call for SLC notices that pass the allowlist. This requires either: (a) adding `zip` to the `UtahLegalsNotice` interface and setting it in the scraper, OR (b) setting it directly in `upsertFromUtahLegals()` when `notice.county === 'salt lake'`.
**Confidence:** MEDIUM — the city will be correct; the zip column may or may not be populated depending on implementation choice.

### Pitfall 5: reCAPTCHA on Detail Pages
**What goes wrong:** Developer tries to fetch detail pages to get full notice text (including better parcel ID extraction). reCAPTCHA blocks automation.
**Why it happens:** The existing scraper comment explicitly documents this: "Does NOT visit individual detail pages because Details.aspx requires reCAPTCHA completion."
**How to avoid:** Stay within the existing design — all data extraction must come from the search results list snippet (300-char limit). The snippet contains the parcel ID label if present. Do not attempt to visit Details.aspx.

### Pitfall 6: "Salt Lake/ Utah" Combined County at Index 18
**What goes wrong:** If the county filter on Utah Legals uses `__doPostBack` differently for the combined "Salt Lake/ Utah" checkbox, it may pull Utah County NODs in addition to Salt Lake County. Provo and Orem NODs would appear in the scrape.
**How to avoid:** Verify county field values in the test run. Utah County notices should show `county="utah"` which will be filtered by `isTargetCounty` (since "utah" is not in TARGET_COUNTIES). The filter already handles this correctly — the county name check is the safety net.

## Open Questions

1. **What label prefix do SLC NOD notices actually use for parcel IDs?**
   - What we know: Real SLC NOD documents exist with format `26-24-406-084-0000`. Branch 2 catches any notice with labels "Serial", "Parcel", "Tax ID", etc. Utah legal notices typically use "Tax Serial Number" or "Serial No." per Utah Code.
   - What's unclear: Whether the 300-char snippet on Utah Legals includes the parcel label + ID or truncates before reaching it.
   - Recommendation: MUST inspect at least one real SLC NOD snippet from a Playwright test run. If all SLC notices log `parcel:none`, the snippet truncates before the parcel number and a different extraction strategy is needed (e.g., fetching more of the snippet or accepting synthetic IDs).

2. **What city values appear in SLC notice `.right` cells?**
   - What we know: The existing parser extracts city from `City:\s*([^\n]+)` in the hidden `.right` cell. For rural counties this reliably returns city names.
   - What's unclear: Whether SLC notices use neighborhood names ("Rose Park"), city name ("Salt Lake City"), or sometimes leave city empty.
   - Recommendation: Log all unique city values from SLC notices in the first test run. Adjust allowlist based on observed values.

3. **Does index 17 correspond to "Salt Lake" (county only) or "Salt Lake/ Utah" (combined)?**
   - What we know: WebFetch showed "Salt Lake" at position 17 and "Salt Lake/ Utah" at position 18 in the rendered list.
   - What's unclear: Whether the rendered text order exactly matches the DOM checkbox `ctl00_ContentPlaceHolder1_as1_lstCounty_{N}` index. The existing file comment shows Carbon=3, Emery=7, etc. — these skip 0-indexed positions for non-county entries (e.g. "Any" at 0 is not counted, or "Davis"=5 occupies that slot).
   - Recommendation: In the test run, print the county values from results when index 17 is checked. If county values include "utah" in addition to "salt lake", switch to index 18 — and verify which index gives Salt Lake County only.

4. **Will the 300-char snippet contain enough data for zip-area validation?**
   - What we know: Snippets truncate at 2000 chars in the code (`snippet.substring(0, 2000)`). Rose Park addresses often don't have the zip visible in the opening text of a trustee sale notice.
   - What's unclear: Whether zip code appears anywhere in the snippet for SLC notices.
   - Recommendation: City-name allowlist is the pragmatic approach for phase 25.5. Post-pilot, if false-positives appear (Sandy/Midvale slipping through as "Salt Lake City"), tighten with address range checks.
