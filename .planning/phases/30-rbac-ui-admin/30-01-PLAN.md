**`admin/layout.tsx`** — server component layout. Does NOT gate the entire `/admin/*` subtree at the layout level (because Brian wants `/admin/audit` URL-accessible to anyone who knows it). Layout just renders shared chrome (back link, page title slot).

**Per-route gating instead:**
- `admin/users/page.tsx` — call `if (!sessionCan(session, "user.manage")) notFound();` at the top of the server component.
- `admin/audit/page.tsx` — NO URL-level gate. Owner-only nav link in the sidebar/bottom-nav handles discoverability. Page reads the audit log; data is sensitive but read-only, and the team is small + trusted.

This is Brian's explicit choice (2026-04-28). If team scales beyond ~5 trusted users, revisit.

**`admin/users/page.tsx`** — server component:
- Fetch all users (active + inactive) with their roles + created_at.
- Render `<UsersTable users={users} />` (client component).
- Render `<NewUserForm />` above the table.

**`UsersTable`** (client component):
- Columns: email, name, roles (multi-select with checkboxes per role; click → `updateUserRoles` server action), is_active toggle (click → `setUserActive`), created, last login (if we track it; defer otherwise), actions: "Reset password" / "Deactivate".
- "Reset password" calls a new admin-only server action that internally invokes the same logic as scripts/admin-reset-password.ts (generate token, send Resend email if RESEND_API_KEY is set, otherwise return URL for manual sharing).

**`NewUserForm`** (client component):
- Fields: email (must end @no-bshomes.com — validate client-side AND server-side), name, roles checkboxes (default empty — Owner explicitly grants).
- Submit calls `createUser` server action which:
  - Validates domain
  - Generates a temporary 32-byte random password (bcrypt-hashed)
  - Inserts the user
  - Triggers a password-reset email so the new user sets their own password immediately
  - Audit log row with action='user.created'
- After success, redirect/refresh.

**`admin-actions.ts`** — new file with all owner-only server actions. Each gated with `userCan(roles, 'user.manage')`. Each audit-logged.

When deactivating a user with active deal/lead assignments: the deactivate succeeds but the auto-fill fall-through (Task 3) handles new transitions. Existing assignments stay in place pointing to the now-inactive user — Owner manually reassigns via the Team panel. (This is intentional: don't auto-cascade. Brian sees the deactivated assignee in the Team panel and can choose where to move the work.)
