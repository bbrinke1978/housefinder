         - Subject: "[Signer Name] has signed — your countersignature is needed"
         - Body: "[signerName] has signed the [contract type] for [address]. Click below to review and countersign. [COUNTERSIGN NOW button]. This link expires in 72 hours."

       - buildExecutedContractHtml(params): Returns HTML email with:
         - Subject: "Fully executed: [Contract Type] for [Property Address]"

    2. Wire email sends into contract-actions.ts (update the sendForSigning and advanceContractStatus functions created in Plan 01):

       - In sendForSigning: After updating status to "sent" and setting tokenExpiresAt, fetch mail settings from scraperConfig (use the existing MAIL_SETTINGS_KEYS pattern from Phase 12). Create Resend instance. Send signing invitation email to first signer using buildSigningInvitationHtml. The signing link URL is `${process.env.NEXTAUTH_URL}/sign/${signer.signingToken}`.

       - In advanceContractStatus when first signer completes: Send countersign notification to second signer using buildCountersignNotificationHtml with their signing token link.

       - In advanceContractStatus when both signers have signed (contract becomes executed):
         (a) Generate final PDF with audit trail via generateContractPdf(contract)
         (b) Upload to blob storage via uploadContract(buffer, `${dealId}/${contractId}-executed.pdf`)
         (c) Update contract row with signedPdfBlobName and documentHash
         (d) Send executed contract email to BOTH signers with the PDF attached (Resend supports attachments: { filename: "contract.pdf", content: buffer })
         (e) Auto-advance deal status: UPDATE deals SET status = 'under_contract' WHERE id = dealId AND status NOT IN ('closed', 'dead')
         (f) Auto-stop campaign: UPDATE campaign_enrollments SET status = 'stopped', stop_reason = 'contract_executed' WHERE property_id = (SELECT property_id FROM deals WHERE id = dealId) AND status = 'active'. Use the existing DB pattern (direct drizzle update, not calling enrollment-actions to avoid circular imports).
         (g) revalidatePath for deal detail and contracts pages

       - Email sends are non-fatal (wrap in try/catch) — contract status still advances even if email delivery fails (consistent with Phase 12 pattern where logContactEvent after email is non-fatal).
