<task type="auto">
  <name>Task 2: getActivityFeed unified query</name>
  <files>app/src/lib/activity-queries.ts</files>
  <action>
New file. Exports:
- `getActivityFeed(propertyId): Promise<ActivityEntry[]>` — query each source, normalize to a common ActivityEntry shape, sort by occurredAt desc, cap at 100.
- `getLastActivity(propertyId): Promise<ActivityEntry | null>` — just the most recent (for card indicator).
- `getActivityCount(propertyId): Promise<number>`.

The ActivityEntry shape:
```typescript
type ActivityEntry = {
  id: string;
  source: 'contact_event' | 'lead_note' | 'deal_note' | 'audit' | 'photo_upload' | 'contract_generated' | 'skip_trace';
  type: string;          // human-readable: 'call', 'email', 'note', 'status_changed', 'photo_added', etc.
  occurredAt: Date;
  actorUserId: string | null;
  actorName: string | null;
  description: string;   // pre-formatted: "Brian called — answered, left voicemail"
  body: string | null;   // long-form note text or diff JSON
  metadata?: Record<string, unknown>;  // e.g. { outcome: 'voicemail' } for filtering
};
```

Source queries (all keyed on propertyId via the linked lead_id):

1. **contact_events** — JOIN leads to filter by propertyId; LEFT JOIN users for actorName; map eventType→type, notes→body, outcome→metadata
2. **lead_notes** — same JOIN to leads; type='note' or 'status_changed' depending on noteType column
3. **deal_notes** — JOIN deals to filter by propertyId; type='note' or 'status_changed'
4. **audit_log** — filter to material actions: action IN ('deal.terms_updated', 'lead.status_changed', 'deal.assignee_changed', 'deal.status_changed', 'property.address_edited', 'lead.assignee_changed'); JOIN entityId to deals/leads to filter by propertyId; build description from oldValue/newValue diff
5. **propertyPhotos** — filter by propertyId; type='photo_added'; description "{actorName} uploaded {n} photo(s)" (group by uploadedBy + day)
6. **dealContracts** — JOIN deals; type='contract_generated'
7. **owner_contacts** (skip-trace) — filter by propertyId, source LIKE 'tracerfy%'; one entry per skip-trace run

Use a single function that runs each source query in parallel, concats, sorts. Don't use Postgres UNION (Drizzle's UNION composer is awkward for this many sources with different shapes — JS sort post-hoc is fine at our volumes).

`getLastActivity` and `getActivityCount` can call `getActivityFeed` and slice — no need for separate optimized queries at our scale. (TODO note in the file: if a property accumulates 1000+ events, switch to source-specific aggregations.)

Always set `actor_user_id = session.user.id`. logAudit() with action='activity.logged' and entity references.

Pick the simple approach for first pass. If dashboard load slows noticeably, switch to LATERAL.
