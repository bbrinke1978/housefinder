  <name>Task 3: Build the feedback form component (used in modal + page)</name>
  <files>app/src/components/feedback/feedback-form.tsx</files>
  <action>
Client component. Props: `{ urlContext?: string; browserContext?: string; onSuccess?: () => void; defaultValues?: Partial<FormValues> }`.

Form fields (use shadcn Form + react-hook-form pattern if already in use; otherwise plain controlled inputs):
- `type` (Select: bug/feature/idea/question, default 'bug')
- `title` (Input, required, max 200)
- `description` (Textarea, optional, max 10,000 chars, helper text "Markdown supported")
- `priority` (Select: low/medium/high/critical, default 'medium')
- `propertyId` (optional combobox; defer to a simple text input for v1 — Brian can paste a URL or ID. Plan 04 adds a property picker.)
- `dealId` (same as propertyId — text input v1)
- Hidden: `urlContext`, `browserContext`

**Paste-from-clipboard handler** on the description textarea:
```typescript
useEffect(() => {
  const el = textareaRef.current;
  if (!el) return;
  const handler = (e: ClipboardEvent) => { /* per RESEARCH.md section 3 */ };
  el.addEventListener("paste", handler);
  return () => el.removeEventListener("paste", handler);
}, []);
```

**Pending-images preview row** below the textarea: shows thumbnail, file name, size, X button to remove. Up to 5 images.

**File picker** button next to the textarea: `<input type="file" accept="image/*" multiple>`, processes the same way as paste (size validation, preview).

**On submit:**
1. `await createFeedbackItem(values)` → returns `{ id }`.
2. For each pending image: `await fetch('/api/feedback/' + id + '/attachments', { method: 'POST', body: formData })`. Show progress as items upload.
3. If `onSuccess` prop provided (modal mode): call it. Otherwise: `router.push('/feedback/' + id)`.
4. On error: surface toast/alert. Item may already be created (partial success); link the user to it.

Loading states + disable Submit while in flight. Use existing toast pattern (sonner or whatever's in use).

**page.tsx** is a server component that:
1. `await auth()` — redirect to /login if no session.
2. Reads search params: `status`, `type`, `priority`, `assigneeId`, `q` (search), `mine` (true→reporterId or assigneeId = me).
3. `await listFeedbackItems(filters)` to fetch the rows.
4. Renders `<FeedbackList items={items} filters={filters} />` (client component for interactive filters) and a "New" button linking to `/feedback/new`.

**feedback-list.tsx** is the client component:
- Filter controls: chips/pills for status (multi-select), type, priority. Search box (debounced, updates URL via router.replace).
- Result table or card grid: title (link to detail), type badge, status badge, priority badge, reporter avatar+name, comment count, attachment count, created relative time ("2h ago").
- Empty state: "No feedback yet. Click New to create the first item."
- Mobile-friendly: collapse to single-column cards under md breakpoint.
- "Mine" toggle: when on, adds `?mine=true` and the server filter scopes to assigneeId or reporterId = current user.

Sort the result client-side or server-side per RESEARCH.md spec (open-first by status, then priority desc, then created_at desc). Server-side preferred to avoid pagination issues later.

URL state pattern: filter changes update query params, server component re-fetches on next render. Mirrors how the existing dashboard filter works (look at `app/src/app/(dashboard)/page.tsx` or wherever filters are wired).
