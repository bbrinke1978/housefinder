## 4. Cross-Phase Regressions

### Did Phase 28 (feedback) break Phase 25/26 (Rose Park) flows?

**No.** Phase 28 is purely additive (new tables, new routes, new components). It shares no code paths with the scraper or Rose Park dashboard queries. The only intersection is the `audit_log` table -- and Phase 28 writes its own `feedback.*` actions there without affecting existing actions.

### Does the v1.4 Phase 31 unified activity feed accidentally hide feedback events?

**The activity feed deliberately scopes feedback events out -- by design.** Verified in `app/src/lib/activity-queries.ts:80-87`:

The `MATERIAL_AUDIT_ACTIONS` allowlist contains exactly: `deal.terms_updated`, `lead.status_changed`, `deal.assignee_changed`, `deal.status_changed`, `property.address_edited`, `lead.assignee_changed`.

The unified property activity feed (`getActivityFeed`, `getDashboardActivityCards`) joins `audit_log` filtered to this allowlist. `feedback.created`, `feedback.status_changed`, and `feedback.deleted` are **intentionally excluded** because feedback items are not lead/property/deal events -- they live in the parallel `feedback_activity` table which powers the per-item timeline (`FeedbackActivityTimeline` rendered inside `feedback-detail.tsx`).

**Verdict:** This is correct behavior, not a bug. Feedback events are not "chatty actions silently dropped" -- they belong to a different feed surface. NO ACTION REQUIRED.

### Does the floating Report button render on /feedback/* itself?

**No -- it self-hides.** Verified in `floating-report-button.tsx:21`: `if (pathname.startsWith("/feedback")) return null;`

This protects against the duplicate-form scenario when a user is already on the feedback list, the new-form page, or the detail page. -- **CORRECT**

### Did Phase 26 deferred 26 of 30 row gap break the dashboard?

**No.** The 26 un-enriched rows still render -- they just have NULL for `building_sqft`, `year_built`, `assessed_value`, `lot_acres`. The detail page Assessor Data Card is conditionally rendered only when at least one of those fields is non-null (per Plan 26-03 research note), so users see no broken UI -- just no card. The dashboard property grid does not display assessor fields at all, so no impact there.

## 7. Architectural Notes for Future Reference

1. **Path A architectural shift (mid-Phase-25.5 -> external commit):** What started as RP-11 ("84116 zip allowlist filter") evolved during execution into a county-wide SLC neighborhood map. The original 84116-only `rose-park-parcel-allowlist.json` is preserved as a fallback. This is **why** Phase 25 had to add 17 additional SLC neighborhood cities to `DEFAULT_TARGET_CITIES` after the fact (Path A follow-on, 2026-04-17). Not a regression -- an intentional broadening.

2. **The 26 of 30 prefix-mismatch gap is data-source diversity, not a bug.** UGRC `SaltLake_County_Addresses` layer used to build the 84116 allowlist appears to cover only the `08-` district. Real-world Rose Park leads come from Utah Legals NOD scrapes, which carry parcel IDs from districts `14`, `21`, `22`, `33`, `15` as well. v1.4 planned fix is to regenerate the allowlist by querying UGRC address layer with a polygon geofence around the Rose Park boundary instead of relying on a single zip-feature-set.

3. **Activity feed scoping is correct as-designed.** Future engineers should NOT add `feedback.*` to `MATERIAL_AUDIT_ACTIONS` in `activity-queries.ts` -- feedback events have their own dedicated `feedback_activity` table and per-item timeline. The unified property activity feed is for lead/property/deal events only.

4. **Missing 26-VERIFICATION.md is a documentation hole**, not a code defect. The b28e72c ROADMAP commit is the authoritative close-out per Brian manual review of the production dashboard.
